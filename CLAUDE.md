# tans

Tangram-themed habit tracker for iOS. Each day you complete a habit, you earn one of seven tangram pieces; seven days fills out the week's silhouette.

This is the **Expo + React Native** rewrite of [`../habit-loop`](../habit-loop) (a Vite/React/shadcn web app generated by Lovable). The web repo is reference-only ‚Äî no shared code at runtime, but the domain logic, types, and silhouette data were copy-ported.

## Stack

- **Expo SDK 54** + **React Native 0.81** + **React 19** (new arch enabled)
- **expo-router** v6 ‚Äî single screen at [app/index.tsx](app/index.tsx)
- **react-native-reanimated 4** + **Moti** for animations
- **react-native-svg** for the tangram board
- **@react-native-async-storage/async-storage** ‚Äî single key, schema-versioned
- **expo-notifications** ‚Äî daily reminder + evening check-in (foreground handler installed)
- **expo-haptics** ‚Äî `Success` impact on habit completion
- **@expo-google-fonts/fredoka** + **@expo-google-fonts/space-grotesk** ‚Äî display + body fonts
- **@expo/vector-icons** (Ionicons) for icons (replaces lucide-react from web)
- **pnpm** (NOT bun ‚Äî user is new to bun and we kept the new project on the lower-friction tooling)

No backend, no auth, no sync. All state local.

## Game rules (mechanics)

Rule constants live in [lib/store.ts](lib/store.ts) (`MAX_ACTIVE_HABITS=3`, `GRADUATION_WEEKS=9`, `MAX_HOLIDAY_DAYS=7`, `HOLIDAY_DANGER_DAYS=4`) and are mirrored verbatim in [components/GameRules.tsx](components/GameRules.tsx). Most are **derived** from `dayEntries` + `habit.createdAt`, not stored:

- **AND across habits** ‚Äî a day earns a piece only if every active, non-paused habit that existed that day is completed (`isDayEarned`). The board/week-strip read from this.
- **Slots & weekly add** ‚Äî max 3 active habits; at most one new habit per week (`addedHabitThisWeek`, derived from `createdAt` week); and adding requires a fully-earned week (current/last/two-weeks-ago via `completedRecentWeek`). `canAddNewHabit()` / `addHabitBlockReason()` gate the FAB + hint.
- **Graduation** ‚Äî 9 *consecutive elapsed full weeks* (`habitGraduationProgress`) flips a habit to `graduatedAt` + `active:false` via an effect; it then renders in the read-only "Permanent habits" box. The in-progress week never counts until elapsed.
- **Holiday hold (per-habit)** ‚Äî `habit.holiday = { startDate, days }`; `isHabitPausedOn` excludes paused days from earning and from full-week checks (so a hold can't break a streak). Set/cleared via `setHabitHoliday`/`clearHabitHoliday`.
- **Slip ‚Üí drop prompt** ‚Äî `habitSlipped` flags a habit whose previous fully-elapsed week wasn't full (past the creation-week grace); the card shows a non-modal "drop it?" banner. Nothing auto-resets.

## Persistence model

- Single AsyncStorage key: `tans:state:v1`.
- Persisted blob: `{ schemaVersion: 1, data: AppState }`. The version field is the migration hook for future schema changes ‚Äî bump it and write a migration in [lib/store.ts](lib/store.ts).
- Hydration is async; the store exposes a `hydrated` boolean and skips writes until hydration finishes (otherwise we'd clobber real data with the empty default on first render).
- IDs use a non-crypto Math.random UUID v4 helper in [lib/store.ts](lib/store.ts) ‚Äî fine for local-only data.

## Animation conventions

- **Moti** for everything that's a tween/spring on a `View`. Pattern:
  ```tsx
  <MotiView from={{ translateY: 600 }} animate={{ translateY: 0 }} exit={{ translateY: 600 }}
            transition={{ type: 'spring', damping: 22, stiffness: 240 }} />
  ```
- **AnimatePresence** from `moti` (NOT framer-motion) wraps conditionally-rendered transitions.
- Bottom sheets are NOT `@gorhom/bottom-sheet` ‚Äî they're `MotiView` slides on a `KeyboardAvoidingView` with a `Pressable` scrim. If we ever need drag-to-dismiss, swap individual sheets to gorhom; for v1 the simpler approach is consistent.

## Notifications

[lib/notifications.ts](lib/notifications.ts) sets up two repeating local notifications:

- `tans.morning-reminder` at 09:00 ‚Äî "Time to make a piece"
- `tans.evening-checkin` at 19:00 ‚Äî "Evening check-in"

Both schedule via `Notifications.SchedulableTriggerInputTypes.DAILY`. Calling `setupNotifications()` is idempotent ‚Äî it cancels by ID and re-schedules. Triggered from `_layout.tsx` on mount; permission denial is silent (we just don't schedule).

Times are exported as `NOTIFICATION_TIMES` and a future settings screen should read/write them.

## Conventions / gotchas

- StyleSheet + theme.ts only ‚Äî no NativeWind, no Tamagui.
- Apostrophes in `<Text>` content must be the typographic `‚Äô` (U+2019), not `'` ‚Äî the `react/no-unescaped-entities` lint rule fires on the straight one. Bonus: it looks better.
- `getToday()` uses local time, not UTC (deliberate fix from the web version).
- Reanimated must be imported once at app boot ‚Äî `import 'react-native-reanimated'` in [_layout.tsx](app/_layout.tsx).
- `@react-navigation/*`, `expo-image`, `expo-symbols`, `expo-web-browser`, `expo-linking`, `expo-system-ui` are still in `package.json` from the Expo template but unused. Trim opportunistically once the app has stabilized.
